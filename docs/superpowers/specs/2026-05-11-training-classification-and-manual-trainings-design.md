# 题单分类与自定义题单设计

## 1. 目标

为当前应用增加两类能力：

- 让题单不再全部堆在一起，而是可按来源类型与自定义分类分组管理
- 支持创建“自定义题单”，用于记录自己做过的好题，并手动维护题目列表

本次设计解决：

- 题单按 `导入题单` / `自定义题单` 两大类展示
- 每个题单支持一个自定义分类，未填写时归入 `未分类`
- 支持创建空的自定义题单
- 支持向自定义题单手动添加任意平台题目
- 支持从自定义题单删除题目
- 保持导入题单与自定义题单使用同一套题单数据结构

本次不做：

- 多分类归属
- 题单排序、拖拽整理
- 平台题目自动补全
- 导入题单中的题目删除或隐藏
- 题单分类批量编辑、合并、重命名

## 2. 方案选择

### 方案 A：只加题单分类，不做自定义题单

优点：

- 改动最小
- 风险最低

缺点：

- 无法承载“记录自己做过的好题”这一核心需求

### 方案 B：来源类型分组 + 自定义分类 + 自定义题单手动维护

优点：

- 一次满足“整理题单”和“沉淀好题”两个目标
- 现有导入链路可以保留
- 后续扩平台时只需补录入能力，不必重做题单结构

缺点：

- 需要数据库迁移
- 需要新增少量前端交互

### 方案 C：一步到位做多平台自动补全框架

优点：

- 后续支持 `Codeforces`、`AtCoder` 等平台时更规整

缺点：

- 当前阶段明显超出需求
- 会引入不必要的抽象和测试负担

### 最终选择

采用方案 B。

## 3. 核心模型

### 3.1 题单模型

题单继续存储在 `trainings` 表中，在现有字段基础上新增：

- `source_type`
  - `imported`：导入题单
  - `manual`：自定义题单
- `category`
  - 单个自定义分类名称
  - 空值统一视为 `未分类`

题单在界面中的归组顺序固定为：

1. `导入题单`
2. `自定义题单`

每一组下面再按分类展示具体题单。

### 3.2 题目模型

题目继续存储在 `problems` 表中，保留现有完成状态、笔记和标签体系，同时从“默认洛谷题目”扩成“平台化题目”。

新增字段：

- `platform`
  - 平台名称，例如 `Luogu`、`Codeforces`、`AtCoder`
- `problem_key`
  - 平台内题目标识，例如 `P1001`、`1700A`
- `url`
  - 可选原题链接

保留字段：

- `title`
- `difficulty`
- `tags`
- `original_tags`
- `user_tags`
- `hidden_original_tags`
- `completed`
- `note`

说明：

- `pid` 的职责由 `problem_key` 承接
- 渲染层后续统一使用“平台 + 题号”描述题目身份
- 标签编辑、完成状态、笔记逻辑不因题目来源变化而变化

### 3.3 来源能力边界

`导入题单` 与 `自定义题单` 的能力边界如下：

- `导入题单`
  - 可删除整个题单
  - 不可手动添加题目
  - 不可删除题单内题目
- `自定义题单`
  - 可删除整个题单
  - 可手动添加题目
  - 可删除题单内题目

这样可以保证：

- 导入题单仍然表达“来自洛谷原始题单的数据”
- 自定义题单表达“用户自行维护的题目集合”

## 4. 数据迁移设计

### 4.1 `trainings` 表迁移

新增列：

- `source_type TEXT NOT NULL DEFAULT 'imported'`
- `category TEXT NOT NULL DEFAULT '未分类'`

旧数据迁移规则：

- 所有历史题单默认视为 `导入题单`
- 所有历史题单默认归到 `未分类`

### 4.2 `problems` 表迁移

新增列：

- `platform TEXT NOT NULL DEFAULT 'Luogu'`
- `problem_key TEXT NOT NULL DEFAULT ''`
- `url TEXT NOT NULL DEFAULT ''`

旧数据迁移规则：

- `platform = 'Luogu'`
- `problem_key = 旧 pid`
- `url = ''`

兼容策略：

- 在迁移完成前，读取逻辑需要兼容旧列存在但新列未初始化的情况
- 迁移完成后，前端统一读取新结构

### 4.3 新建自定义题单的 ID 生成

自定义题单不能继续依赖洛谷题单 ID。

采用本地整数 ID 方案：

- 读取当前 `trainings` 最大 `id`
- 新建时使用 `max(id) + 1`

约束：

- 只在本地数据库内保证唯一
- 不引入额外 UUID 依赖

这样可以保持当前表结构与 IPC 类型尽量稳定。

## 5. 主进程与 IPC 设计

### 5.1 导入题单

导入流程保留现有抓取链路，但写入时补充：

- `source_type = imported`
- `category = 用户选择值或 '未分类'`

说明：

- 导入弹窗需要新增分类输入
- 若再次导入同一洛谷题单，沿用现有覆盖写入策略，不引入复杂合并逻辑

### 5.2 新建自定义题单

新增 IPC：

- `create-training`

输入：

- `name`
- `description`
- `category`

固定写入：

- `source_type = manual`
- `problem_count = 0`

输出：

- 新建后的题单摘要对象，用于前端刷新或直接插入列表

### 5.3 自定义题单添加题目

新增 IPC：

- `create-problem`

输入：

- `trainingId`
- `platform`
- `problemKey`
- `title`
- `url`
- `difficulty`
- `tags`

约束：

- 仅允许向 `source_type = manual` 的题单添加
- 若目标题单不是自定义题单，直接拒绝

写入规则：

- `tags`、`original_tags` 初始保持一致
- `user_tags`、`hidden_original_tags` 初始为空
- `completed = 0`
- `note = ''`
- 同步更新题单 `problem_count`

### 5.4 自定义题单删除题目

新增 IPC：

- `delete-problem`

约束：

- 仅允许删除 `source_type = manual` 题单中的题目
- 删除后同步更新题单 `problem_count`

### 5.5 查询题单列表

`get-trainings` 返回值新增：

- `sourceType`
- `category`

前端使用这两个字段完成两层分组展示。

## 6. 前端交互设计

### 6.1 侧边栏结构

侧边栏最终结构：

```text
导入题单
  未分类
    题单 A
  图论
    题单 B

自定义题单
  好题
    题单 C
  待补
    题单 D
```

交互要求：

- 分组标题默认展开
- 分类组默认展开
- 无数据时显示空状态提示

### 6.2 顶部操作

保留：

- `+ 导入`

新增：

- `+ 新建`

按钮语义：

- `+ 导入`：创建导入题单
- `+ 新建`：创建自定义题单

### 6.3 导入弹窗

在现有导入弹窗中增加：

- `分类` 输入框

字段：

- 题单链接或 ID
- 分类（可空）

空分类时自动归入 `未分类`。

### 6.4 新建自定义题单弹窗

字段：

- `题单名称`：必填
- `分类`：选填
- `描述`：选填

成功后：

- 关闭弹窗
- 自动选中新题单
- 主区域进入空题单状态

### 6.5 自定义题单空状态

当选中的是空的自定义题单时，主区域显示：

- 当前题单暂无题目
- 一个明显的 `添加题目` 按钮

### 6.6 添加题目弹窗

字段：

- `平台`：必填
- `题号`：必填
- `标题`：必填
- `链接`：选填
- `难度`：选填
- `标签`：选填，使用逗号切分或沿用现有标签输入方式

校验规则：

- `平台`、`题号`、`标题` 为空时不可提交
- `链接` 若填写，不强制联网校验

### 6.7 删除题目交互

仅自定义题单中的题目项显示删除入口。

交互要求：

- 点击删除时弹出确认框
- 明确提示“仅删除当前自定义题单中的该题记录”

### 6.8 标题区提示

主区域标题旁增加来源标记：

- `导入题单`
- `自定义题单`

目的是让用户随时知道当前题单的编辑权限边界。

## 7. 展示与兼容行为

### 7.1 题目展示

题目列表中继续显示：

- 题号
- 标题
- 难度
- 标签
- 完成状态
- 笔记

对于非洛谷平台题目：

- 题号区域显示 `problem_key`
- 若存在 `url`，点击时打开该链接
- 若没有 `url`，则题号和标题不触发外链

### 7.2 标签体系

标签体系继续保留当前设计：

- 原始标签
- 用户补充标签
- 隐藏原始标签

对于手动添加题目：

- 首次写入的标签既作为 `tags`，也作为 `original_tags`
- 后续编辑逻辑与导入题目一致

### 7.3 完成进度

题单完成进度继续按：

- `completedCount / problemCount`

统一计算，不区分导入题单与自定义题单。

## 8. 错误处理

需要显式处理以下场景：

- 新建题单时名称为空
- 添加题目时必填字段为空
- 尝试向导入题单添加题目
- 尝试删除导入题单中的题目
- 数据迁移后旧字段缺失或空值异常

错误原则：

- 主进程拒绝非法写入
- 前端展示明确错误提示
- 不吞掉异常，不静默失败

## 9. 验证方案

### 自动化验证

建议新增主进程数据库测试，覆盖：

- 旧库迁移后题单默认带上 `source_type` 与 `category`
- 新建自定义题单写入正确
- 仅自定义题单可添加题目
- 仅自定义题单可删除题目
- 添加/删除题目后 `problem_count` 正确更新

建议新增前端逻辑测试，覆盖：

- 题单列表按来源类型与分类正确分组
- 新建自定义题单后默认选中

### 手动验证

1. 打开已有数据库，历史题单应全部出现在 `导入题单 -> 未分类`
2. 导入一个新题单并填写分类，应出现在对应导入分类下
3. 新建一个自定义题单，应出现在 `自定义题单` 下
4. 向自定义题单添加一题 `Codeforces` 或任意平台题目，应能正常展示
5. 删除该题目后，题单进度与题目数量应正确回退
6. 导入题单内不应出现“添加题目”或“删除题目”入口
7. 现有完成状态、笔记、标签编辑能力不应回归

## 10. 风险与限制

- 当前 `trainings.id` 仍是单整数主键，自定义题单与导入题单共用 ID 空间；虽然足够可用，但后续若引入更多外部数据源，需要继续关注 ID 生成策略
- 手动题目的 `difficulty`、`tags` 全部依赖用户输入，本次不保证跨平台一致性
- 若后续要支持“导入题单删题但保留原始信息”，应新增“隐藏题目”状态，而不是复用真实删除
- 若后续要支持平台自动补全，应在当前字段模型上补抓取逻辑，不需要再改题单结构
