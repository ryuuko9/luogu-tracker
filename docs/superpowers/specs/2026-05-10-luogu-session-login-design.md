# 洛谷会话登录导入设计

## 1. 目标

为当前应用增加“应用内登录洛谷”的能力，使需要登录权限的题单可以直接导入，无需用户手动复制 Cookie。

本次设计解决：

- 应用内弹出洛谷登录窗口
- 登录状态持久化到本机 Electron 会话
- 导入题单时自动复用该会话
- 侧边栏在登录后显示头像
- 支持退出登录与重新登录

本次不做：

- 多账号并存
- 系统级账号托管
- 自动处理验证码或更复杂风控

## 2. 方案选择

### 方案 A：手动粘贴 Cookie

优点：

- 实现快

缺点：

- 依赖用户正确复制 Cookie
- 易失效
- 容易拿不到完整字段

### 方案 B：应用内登录窗口 + 持久会话

优点：

- 不依赖手动 Cookie
- 登录后重启应用通常仍能保持会话
- 导入受限题单体验最自然

缺点：

- 主进程实现更复杂

### 最终选择

采用方案 B。

## 3. 主体设计

### 3.1 会话模型

使用固定 Electron 持久会话：

- `persist:luogu-auth`

所有洛谷登录窗口与需要鉴权的抓取请求都复用该会话。

### 3.2 登录窗口

主进程创建单独的 `BrowserWindow` 打开：

- `https://www.luogu.com.cn/auth/login`

窗口行为：

- 若已有登录窗口，则直接聚焦
- 登录成功后自动关闭
- 关闭后不影响主窗口

### 3.3 登录状态判断

登录状态不再依赖手动 Cookie 文本，而是依赖会话本身。

判断流程：

1. 从会话 Cookie 中读取 `_uid`
2. 若无 `_uid`，视为未登录
3. 若有 `_uid`，用同一会话请求 `https://www.luogu.com.cn/user/setting`
4. 若返回 `401` 或页面上下文显示 `needLogin`，视为未登录
5. 若访问成功，再请求公开用户页 `https://www.luogu.com.cn/user/<uid>` 获取头像和昵称

## 4. 数据流

### 4.1 登录

1. 用户点击侧边栏登录入口
2. 主进程打开洛谷登录窗口
3. 用户在窗口内完成登录
4. 会话写入本机持久 session
5. 主进程重新读取登录状态并返回给前端
6. 前端更新头像显示

### 4.2 导入受限题单

1. 用户点击导入
2. `scrapeTraining` 复用 `persist:luogu-auth` 对应的 session 发请求
3. 若会话有效，题单正常解析导入
4. 若会话失效，返回“请重新登录洛谷”

### 4.3 退出登录

1. 用户点击头像
2. 打开登录管理弹窗
3. 用户点击退出登录
4. 主进程清理该持久 session 的 Cookie 和站点存储
5. 前端状态恢复为未登录

## 5. 前端交互

### 5.1 侧边栏状态展示

采用方案 B：

- 未登录时显示简洁登录按钮
- 已登录时显示头像

### 5.2 已登录后的操作

点击头像打开一个轻量管理弹窗，展示：

- 头像
- 昵称
- UID
- `重新登录`
- `退出登录`

### 5.3 未登录时的受限题单导入提示

- 若导入受限题单但未登录，提示：`该题单需要登录后访问，请先登录洛谷`
- 若会话失效，提示：`当前洛谷登录状态已失效，请重新登录`

## 6. 抓取链路设计

`scrapeTraining` 改为支持可选 `Session` 参数。

规则：

- 公开请求可走默认网络栈
- 需要统一行为时，直接使用传入的 `Session.fetch`
- 鉴权相关请求全部走洛谷持久会话

## 7. 验证方案

### 自动化验证

补充主进程解析测试：

- 登录错误上下文识别
- 用户页头像与昵称解析

### 手动验证

1. 首次打开应用，状态应为未登录
2. 点击登录，弹出洛谷登录窗口
3. 登录成功后窗口关闭，侧边栏显示头像
4. 导入 `465333` 成功
5. 重启应用后仍显示头像
6. 退出登录后头像消失，再导入受限题单应提示重新登录

## 8. 风险与限制

- 洛谷若调整登录流程，窗口自动关闭时机可能需重新适配
- 若平台触发验证码或风控，仍需人工完成
- 持久化的是本机当前应用会话，不是跨设备同步登录
